## 2. Техника 1: Классы эквивалентности (Equivalence Partitioning)

### 2.1 Теоретическая основа

**Класс эквивалентности** — это множество входных значений, которые, согласно спецификации, должны обрабатываться системой **одинаковым образом**.

**Главный принцип:** Достаточно проверить одно значение из каждого класса. Если тест пройден — вероятно, все значения класса работают корректно.

### 2.2 Алгоритм выделения классов
Шаг 1: Определить входные параметры
↓
Шаг 2: Разбить область допустимых значений на классы
↓
Шаг 3: Выделить классы недопустимых значений (если есть)
↓
Шаг 4: Выбрать по одному представителю из каждого класса

text

### 2.3 Примеры для разных типов полей

| Тип поля | Пример | Классы эквивалентности | Тестовые значения |
|:---|:---|:---|:---|
| **Целое число с диапазоном** | Возраст 0-120 | Валидные: 0-120<br>Невалидные: <0<br>Невалидные: >120 | 50 (валид)<br>-5 (невалид)<br>150 (невалид) |
| **Перечисление** | Тип пользователя: admin, editor, user | 3 класса: admin, editor, user<br>+ класс невалидных значений | 'admin', 'editor', 'user', 'guest' |
| **Строка с ограничением длины** | Имя: 2-50 символов | Валидные: 2-50<br>Невалидные: <2<br>Невалидные: >50<br>Невалидные: спецсимволы | 'Анна'<br>'А'<br>(51 символ)<br>'@#$%' |
| **Булево значение** | Флаг согласия (true/false) | true, false | true, false |
| **Дата** | Дата рождения (01.01.1900 - текущая) | Валидные: 1900-2025<br>Невалидные: до 1900<br>Невалидные: в будущем | '15.06.2000'<br>'01.01.1899'<br>'01.01.2030' |
| **Число с плавающей точкой** | Сумма заказа (0 - 1 000 000) | Валидные: 0-1 000 000<br>Невалидные: <0<br>Невалидные: >1 000 000 | 5000.50<br>-0.01<br>1 000 001 |

### 2.4 Практический пример: Поле «Количество товара» в интернет-магазине

**Требование:** Количество товара от 1 до 99 включительно.

**Классы эквивалентности:**
| Класс | Диапазон | Представитель | Ожидаемый результат |
|:---|:---|:---|:---|
| Валидный | 1 - 99 | 50 | Товар добавлен |
| Невалидный (нижний) | < 1 | 0 | Ошибка «Минимальное количество — 1» |
| Невалидный (верхний) | > 99 | 100 | Ошибка «Максимальное количество — 99» |
| Невалидный (не число) | Не число | 'abc' | Ошибка «Введите число» |

**Итого:** 4 теста вместо 99.

## 3. Техника 2: Граничные значения (Boundary Value Analysis)

### 3.1 Почему границы важны?

**Статистика:** Около 80% дефектов в программах, работающих с диапазонами, находится **на границах значений**. Разработчики часто ошибаются в операторах сравнения (`<` вместо `<=`, `>` вместо `>=`) или в смещении на единицу (off-by-one error).

### 3.2 Правила выбора границ

| Тип границы | Значения для проверки |
|:---|:---|
| Закрытая граница [a, b] | a, a+1, b-1, b |
| Открытая граница (a, b) | a+1, a+2, b-2, b-1 |
| Полуоткрытая [a, b) | a, a+1, b-1, b (b — уже за границей) |
| Бесконечная граница | минимальное/максимальное значение, минимальное-1 |

**Стандартный набор для диапазона от L до R:**
L-1 → сразу за нижней границей (невалид)
L → нижняя граница (валид)
L+1 → сразу после нижней границы (валид)
R-1 → сразу перед верхней границей (валид)
R → верхняя граница (валид)
R+1 → сразу за верхней границей (невалид)

text

### 3.3 Пример: Тестирование кредитного калькулятора

**Требование:** Сумма кредита от 10 000 до 5 000 000 рублей.

| Значение | Статус | Ожидаемый результат |
|:---|:---|:---|
| 9 999 | Ниже границы | Ошибка «Минимальная сумма — 10 000» |
| 10 000 | Граница | Валидно |
| 10 001 | Сразу после границы | Валидно |
| 2 500 000 | Типичное значение | Валидно |
| 4 999 999 | Перед верхней границей | Валидно |
| 5 000 000 | Верхняя граница | Валидно |
| 5 000 001 | Выше границы | Ошибка «Максимальная сумма — 5 000 000» |

### 3.4 Особые случаи граничных значений

| Ситуация | Как тестировать | Пример |
|:---|:---|:---|
| **Пустая строка** | Проверить строку длиной 0 | `name = ""` |
| **Максимальная длина строки** | Проверить L, L+1 | `name = "A" x 255`, `"A" x 256` |
| **Ноль и отрицательные числа** | Проверить 0, -1, -0.01 | `quantity = 0`, `quantity = -1` |
| **Дата 29 февраля** | Проверить високосный и невисокосный год | `29.02.2024` (високосный), `29.02.2023` (ошибка) |
| **Максимальное число в БД** | Проверить int(11) → 2 147 483 647 и +1 | `price = 2147483647`, `price = 2147483648` |

## 4. Техника 3: Таблицы принятия решений (Decision Tables)

### 4.1 Когда нужны таблицы решений

Таблицы решений применяются, когда:
- Есть **сложная бизнес-логика** с несколькими условиями
- Результат зависит от **комбинаций условий**
- Нужно выявить **пропущенные правила** (неучтённые комбинации)

### 4.2 Структура таблицы решений
┌────────────────────────────┬─────┬─────┬─────┬─────┐
│ УСЛОВИЯ │ 1 │ 2 │ 3 │ 4 │
├────────────────────────────┼─────┼─────┼─────┼─────┤
│ Условие 1 (Да/Нет) │ Д │ Д │ Н │ Н │
│ Условие 2 (Да/Нет) │ Д │ Н │ Д │ Н │
├────────────────────────────┼─────┼─────┼─────┼─────┤
│ ДЕЙСТВИЯ │ │ │ │ │
│ Действие 1 │ X │ │ │ │
│ Действие 2 │ │ X │ X │ │
└────────────────────────────┴─────┴─────┴─────┴─────┘

text

### 4.3 Алгоритм построения
Шаг 1: Выделить все условия (входные параметры с булевыми значениями)
↓
Шаг 2: Выделить все действия (результаты)
↓
Шаг 3: Построить таблицу всех комбинаций условий (2^n строк)
↓
Шаг 4: Для каждой комбинации определить ожидаемые действия
↓
Шаг 5: Упростить таблицу (удалить недостижимые и дублирующие правила)

text

### 4.4 Пример 1: Скидка в интернет-магазине

**Условия:**
1. Сумма заказа > 5000? (Да/Нет)
2. Пользователь — VIP? (Да/Нет)
3. Сегодня чёрная пятница? (Да/Нет)

**Действия:**
- A. Скидка 10%
- B. Скидка 5%

**Полная таблица (2^3 = 8 правил):**

| Правило | 1 | 2 | 3 | 4 | 5 | 6 | 7 | 8 |
|:---|:---|:---|:---|:---|:---|:---|:---|:---|
| **Условия** |
| Сумма > 5000 | Д | Д | Д | Д | Н | Н | Н | Н |
| VIP клиент | Д | Д | Н | Н | Д | Д | Н | Н |
| Чёрная пятница | Д | Н | Д | Н | Д | Н | Д | Н |
| **Действия** |
| Скидка 10% | X | X | X |   | X |   |   |   |
| Скидка 5% |   |   |   |   |   | X | X |   |
| Беспл. доставка | X |   | X |   | X |   |   |   |

**Упрощение (убираем дублирование):**

| Правило | A | B | C | D | E |
|:---|:---|:---|:---|:---|:---|
| Сумма > 5000 | Д | Д | Д | Н | Н |
| VIP клиент | Д | Д | Н | Д | Н |
| Чёрная пятница | Д | Н | (любое) | Д | (любое) |
| **Скидка** | 10% | 10% | 10% | 10% | 5% |
| **Доставка** | Беспл. | Платная | Беспл. | Беспл. | Платная |

**Тест-кейсы на основе таблицы:**
1. A: Сумма > 5000, VIP, ЧП → скидка 10%, доставка бесплатно
2. B: Сумма > 5000, VIP, не ЧП → скидка 10%, доставка платная
3. C: Сумма > 5000, не VIP → скидка 10%, доставка бесплатно (ЧП не важно)
4. D: Сумма ≤ 5000, VIP, ЧП → скидка 10%, доставка бесплатно
5. E: Сумма ≤ 5000, не VIP → скидка 5%, доставка платная

## 5. Техника 4: Диаграммы переходов состояний (State Transition Diagrams)

### 5.1 Основные понятия

| Понятие | Определение | Пример |
|:---|:---|:---|
| **Состояние** | Устойчивое положение системы | «Заказ создан», «Оплачен», «Доставлен» |
| **Переход** | Движение из одного состояния в другое | Нажатие «Оплатить» |
| **Событие** | Триггер перехода | Поступление денег на счёт |
| **Действие** | Что происходит при переходе | Списание средств, отправка письма |

### 5.2 Символы диаграммы состояний
Состояние: [ Состояние А ]
│
Событие/Действие │ (событие)
↓
Состояние: [ Состояние Б ]

Начальное состояние: ([]) → [ Состояние ]
Конечное состояние: [ Состояние ] → ([])

**Диаграмма:**

│ │
↓ │

│ │ │
│ (отмена) │ (возврат) │ (возврат)
↓ ↓ ↓
[ОТМЕНЁН] ←────────────┘ │
→ ([]) │
↓
[ВОЗВРАТ]
→ ([])

text

**События и переходы:**

| Текущее состояние | Событие | Новое состояние | Действие |
|:---|:---|:---|:---|
| КОРЗИНА | Оформить заказ | ОЖИДАЕТ ОПЛАТЫ | Заблокировать товар |
| ОЖИДАЕТ ОПЛАТЫ | Оплата получена | ОПЛАЧЕН | Списать деньги |
| ОЖИДАЕТ ОПЛАТЫ | Отмена (24 часа) | ОТМЕНЁН | Разблокировать товар |
| ОПЛАЧЕН | Передать в доставку | ОТПРАВЛЕН | Отправить уведомление |
| ОПЛАЧЕН | Запросить возврат | ВОЗВРАТ | Заблокировать возврат |
| ОТПРАВЛЕН | Подтвердить доставку | ДОСТАВЛЕН | Завершить заказ |
| ОТПРАВЛЕН | Возврат при отправке | ВОЗВРАТ | Вернуть деньги |

### 5.4 Как тестировать по диаграмме состояний

**Методика:**

1. Построить диаграмму всех возможных состояний и переходов
2. Для каждого перехода создать тест-кейс
3. Проверить **негативные сценарии** (невозможные переходы)
4. Проверить **короткие пути** (прыжок через несколько состояний, если это запрещено)

**Чек-лист тестов для диаграммы состояний:**

| Тип теста | Пример |
|:---|:---|
| **Счастливый путь** | КОРЗИНА → ОЖИДАЕТ ОПЛАТЫ → ОПЛАЧЕН → ОТПРАВЛЕН → ДОСТАВЛЕН |
| **Сброс (отмена)** | ОЖИДАЕТ ОПЛАТЫ → ОТМЕНЁН |
| **Возврат** | ОПЛАЧЕН → ВОЗВРАТ |
| **Негативный (запрещённый переход)** | ОТМЕНЁН → ОПЛАЧЕН (попробовать оплатить отменённый заказ) |
| **Повторный переход** | ОПЛАЧЕН → ОПЛАЧЕН (повторная оплата) |
| **Переход из недопустимого состояния** | Попытка отправить неоплаченный заказ |

## 6. Комбинирование техник: когда что использовать

### 6.1 Матрица выбора техник

| Тип системы/функции | Основная техника | Дополнительная техника |
|:---|:---|:---|
| **Калькулятор, математические функции** | Граничные значения | Эквивалентное разделение |
| **Форма ввода с валидацией** | Эквивалентное разделение | Граничные значения |
| **Бизнес-логика с правилами** | Таблица решений | Попарное тестирование |
| **Workflow, статусы, жизненный цикл** | Диаграмма переходов | Таблица решений |
| **Фильтры и поиск** | Попарное тестирование | Эквивалентное разделение |
| **API с большим количеством параметров** | Попарное тестирование | Граничные значения |

### 6.2 Пример комбинирования: Тестирование функции перевода денег

**Функция:** Перевод средств между счетами.

**Параметры:**
- Сумма перевода (число)
- Валюта (RUB, USD, EUR)
- Свой счёт (true/false)
- Наличие комиссии (true/false)

**Применяем техники:**

| Техника | Применение | Количество тестов |
