## 2. Тест-стратегия и тест-план

### 2.1 Тест-стратегия (Test Strategy)

**Тест-стратегия** — это документ высокого уровня, который описывает **принципы и подходы** к тестированию на всём проекте. Она **не зависит от конкретного релиза** и может использоваться для всего продукта.

#### Что входит в тест-стратегию:

| Раздел | Описание | Пример |
|:---|:---|:---|
| **Цели тестирования** | Чего хотим достичь | Обнаружить 95% критических дефектов до релиза |
| **Уровни тестирования** | Что тестируем на каждом этапе | Модульное (разработчики), интеграционное (QA), системное (QC), приёмочное (заказчик) |
| **Подходы** | Какие методы используем | Чёрный ящик для UI, белый для юнит-тестов, серый для API |
| **Типы тестирования** | Функциональное, нагрузочное, безопасность, удобство использования | Нагрузочное только для платёжного шлюза |
| **Критерии входа/выхода** | Когда начинаем и заканчиваем тестирование | Вход: стабильная сборка; выход: нет критических багов |
| **Роли и ответственность** | Кто что делает | Разработчик пишет юнит-тесты, QA — API-тесты |
| **Инструменты** | Какие инструменты используем | PHPUnit, Laravel Dusk, Postman, Jira |
| **Метрики** | Как измеряем прогресс | Покрытие кода > 80%, количество открытых багов < 10 |

#### Пример тест-стратегии для учебного проекта (интернет-магазин на Laravel):

```markdown
# Тест-стратегия проекта ShopOnline

## 1. Цели
- Обеспечить бесперебойную работу ключевых сценариев: регистрация, поиск, добавление в корзину, оплата.
- Обнаружить 100% дефектов с критическим приоритетом до релиза.

## 2. Уровни и подходы
- Модульное тестирование (Unit): белый ящик, PHPUnit
- Интеграционное тестирование: серый ящик, PHPUnit + БД
- API-тестирование: серый ящик, Postman + автоматизация
- UI-тестирование: чёрный ящик, Laravel Dusk
- Приёмочное тестирование: чёрный ящик, ручное (заказчик)

## 3. Типы тестирования
- Функциональное: все сценарии
- Нагрузочное: только корзина и оплата (1000 пользователей)
- Безопасность: формы входа и регистрации (SQL-инъекции, XSS)
- Регрессионное: после каждого релиза

## 4. Критерии
- Вход в тестирование: сборка прошла все юнит-тесты
- Выход: нет критических багов, покрытие кода ≥ 75%

## 5. Инструменты
- PHPUnit, Laravel Dusk, Postman, Jira, GitHub Actions
2.2 Тест-план (Test Plan)
Тест-план — это документ для конкретного релиза или этапа тестирования. Он детализирует, кто, когда и что конкретно будет делать.

Стандартный шаблон тест-плана (по IEEE 829)
markdown
# Тест-план Проект ShopOnline / Релиз 2.0 / Спринт 5

## 1. Идентификатор документа
TP-SHOP-2025-04-01

## 2. Объекты тестирования
- Корзина покупок
- Оформление заказа
- Платёжный шлюг (интеграция с банком)

## 3. Что НЕ тестируем
- Ручное создание заказа в админке
- Мобильное приложение (тестируется отдельно)

## 4. Подходы
- Функциональное тестирование (чёрный + серый ящик)
- Регрессионное тестирование (автоматизированное, 80% тестов)
- Smoke-тестирование перед каждым билдом

## 5. Критерии входа/выхода
**Вход:**
- Функционал передан в тестирование
- Юнит-тесты успешно пройдены
- Тест-кейсы готовы (≥ 90%)

**Выход:**
- Все запланированные тесты выполнены
- Нет багов с приоритетом Critical и High
- Отчёт о тестировании согласован с PM

## 6. Ресурсы
**Команда:**
- QA-инженер: Иванов И. (ручное тестирование)
- Автоматизатор: Петров П. (API и E2E)

**Окружение:**
- Стенд: staging (http://staging.shoponline.local)
- БД: копия продуктивной с анонимизацией

## 7. Расписание
| Действие | Дата | Ответственный |
|:---|:---|:---|
| Smoke-тесты | 01.04 | Иванов И. |
| Функциональное тестирование | 02-05.04 | Иванов И. |
| Регрессионные тесты | 06-07.04 | Петров П. |
| Нагрузочное тестирование | 08.04 | Петров П. |
| Отчёт | 09.04 | Иванов И. |

## 8. Риски и митигации
| Риск | Вероятность | Действие |
|:---|:---|:---|
| Задержка передачи сборки | Средняя | Сдвиг расписания |
| Баг в платёжном шлюзе | Высокая | Подготовить мок-объекты |
Упражнение 1 (5 минут)
Задание: Для вашего учебного проекта (например, блог на Laravel) напишите краткую тест-стратегию (3-5 пунктов) и тест-план для ближайшего спринта.

3. Тест-дизайн: искусство создания эффективных тестов
Тест-дизайн — это этап, на котором мы превращаем требования в тестовые сценарии. Хороший тест-дизайн находит максимум дефектов при минимуме тест-кейсов.

3.1 Основные техники тест-дизайна
Техника	Суть	Когда применять	Пример
Эквивалентное разделение	Разбиваем входные данные на классы, внутри которых поведение одинаковое	Поля с диапазонами, переключатели	Возраст: 0-17 (детский), 18-64 (взрослый), 65+ (пенсионный). Достаточно одного теста из каждого класса
Анализ граничных значений	Проверяем значения на границах классов и сразу за ними	Там, где есть «края»	Для 0-17: проверяем 17, 18, 0, 1, -1
Попарное тестирование	Комбинируем значения параметров так, чтобы каждая пара значений встретилась хотя бы раз	Много параметров с несколькими значениями	Браузер (Chrome, Firefox) × ОС (Win, Mac) × Роль (Гость, Админ) → 2×2×2 = 8 тестов, попарно даёт 4
Таблица принятия решений	Строим таблицу «условия → действия»	Бизнес-правила со сложной логикой	Если клиент VIP И сумма > 5000 ИЛИ есть промокод → доставка бесплатно
State-переходы	Тестируем переходы между состояниями системы	Приложения с состояниями (заказ: создан → оплачен → отправлен)	Логин: начальное → ввод логина → ожидание → успешно/ошибка
Предугадывание ошибки	Используем опыт, чтобы предположить, где ошибки	Всегда, особенно на критических модулях	Проверить поля с датами на 29 февраля, 31 ноября, пустые строки
3.2 Подробный разбор техник с примерами
Техника 1: Эквивалентное разделение (Equivalence Partitioning)
Принцип: Все значения из одного класса эквивалентны — если дефект есть, он проявится на любом значении из этого класса.

Пример для метода calculateDiscount($total, $status):

php
// Входные параметры:
// $total: от 0 до 1 000 000
// $status: 'new', 'regular', 'vip'

// Выделяем классы эквивалентности:

// Для $total:
// Класс 1: 0 (минимальное значение)
// Класс 2: 1 – 999 (маленькая сумма)
// Класс 3: 1000 – 50000 (средняя сумма)
// Класс 4: 50001 – 1 000 000 (большая сумма)
// Класс 5: > 1 000 000 (невалидное)

// Для $status:
// Класс 1: 'new'
// Класс 2: 'regular'
// Класс 3: 'vip'
// Класс 4: 'unknown' (невалидное)

// Минимальное количество тестов: 5 (total) × 4 (status) = 20
// Но если поведение одинаковое для всех сумм из одного класса, то достаточно 4×4 = 16
Ваш тест-кейс (один из комбинаций):

text
Название: Total из класса "маленькая сумма" + статус "vip"
Вход: $total = 500, $status = 'vip'
Ожидаемый результат: скидка 15%, итог = 425
Техника 2: Анализ граничных значений (Boundary Value Analysis)
Принцип: Ошибки чаще всего возникают на границах диапазонов.

Пример для бесплатной доставки (условие: if ($afterDiscount > 5000)):

Граница	Значение	Ожидаемая доставка	Почему важно
Нижняя граница	5000.00	Платная (300)	> а не >=
Сразу за границей	5000.01	Бесплатная	Первое значение с бесплатной доставкой
Типичное значение	10000	Бесплатная	Для сравнения
Минимальное значение	0	Платная	Граница диапазона
Отрицательное	-100	Ошибка	Невалидный ввод
Test-кейсы для граничных значений:

text
1. afterDiscount = 5000.00 → доставка = 300
2. afterDiscount = 5000.01 → доставка = 0
3. afterDiscount = 0.00 → доставка = 300
4. afterDiscount = -0.01 (если возможно) → ошибка
Техника 3: Таблица принятия решений (Decision Table)
Принцип: Для сложной бизнес-логики строим таблицу, где строки — комбинации условий, столбцы — действия.

Условия:

Пользователь авторизован (ДА/НЕТ)

Сумма заказа > 5000 (ДА/НЕТ)

Есть активный промокод (ДА/НЕТ)

Действия:

A. Применить скидку 10%

B. Бесплатная доставка

C. Отправить SMS с подтверждением

Таблица принятия решений:

Правило	1	2	3	4	5	6	7	8
Условия								
Авторизован	ДА	ДА	ДА	ДА	НЕТ	НЕТ	НЕТ	НЕТ
Сумма > 5000	ДА	ДА	НЕТ	НЕТ	ДА	ДА	НЕТ	НЕТ
Промокод	ДА	НЕТ	ДА	НЕТ	ДА	НЕТ	ДА	НЕТ

Скидка 10%	ДА	НЕТ	НЕТ	НЕТ	ДА	НЕТ	НЕТ	НЕТ

Тест-кейсы по правилам (достаточно 4-5 из 8):

Правило 1: Авторизован + сумма > 5000 + промокод → скидка 10% + беспл. доставка

Правило 2: Авторизован + сумма > 5000 + нет промокода → беспл. доставка (нет скидки)

Правило 3: Авторизован + сумма ≤ 5000 + промокод → ничего (промокод не работает)

Правило 5: Не авторизован + сумма > 5000 + промокод → скидка 10% + беспл. доставка

Техника 4: Попарное тестирование (Pairwise Testing)
Принцип: Большинство дефектов возникает из-за взаимодействия двух параметров. Тройные и более комбинации редки.

Пример: Тестируем поиск товаров с фильтрами:

Параметр	Значения
Категория	Электроника, Одежда, Книги
Цена	Дешёвые, Средние, Дорогие
Бренд	Apple, Samsung, Nike
Рейтинг	1-2, 3-4, 5
Полный перебор: 3 × 3 × 3 × 3 = 81 тест.
Попарное тестирование: 9 тестов (каждая пара значений встретится хотя бы раз).

3.3 Чек-лист для выбора техники тест-дизайна
Если нужно протестировать...	Используйте технику
Поле с диапазоном значений (возраст, сумма, количество)	Эквивалентное разделение + Граничные значения
Сложную бизнес-логику с несколькими условиями (if-else)	Таблица принятия решений
Систему с состояниями (заказ, пользователь, сессия)	State-переходы
Комбинацию нескольких параметров (фильтры, поиск)	Попарное тестирование
Критический модуль, где опыт подсказывает опасные места	Предугадывание ошибки
