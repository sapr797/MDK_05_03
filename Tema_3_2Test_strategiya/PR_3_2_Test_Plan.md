### Разница между тест-стратегией и тест-планом

| Характеристика | Тест-стратегия | Тест-план |
|:---|:---|:---|
| **Уровень** | Высокий (стратегический) | Детальный (тактический) |
| **Длительность** | На весь проект | На конкретный релиз/спринт |
| **Аудитория** | Руководство, заказчик, PM | Команда тестирования, разработчики |
| **Изменяемость** | Редко | Перед каждым релизом |
| **Что содержит** | Подходы, принципы, инструменты | Даты, задачи, конкретные тесты |
| **Ответственный** | Тест-менеджер / Lead QA | QA-инженер / Тестировщик |

## 4. Подходы к тестированию

### 4.1 Чёрный ящик (UI, API)
- **Применяется для:** F-01, F-02, F-06 (все пользовательские сценарии)
- **Метод:** Тест-кейсы на основе требований, исследовательское тестирование

### 4.2 Белый ящик (Unit)
- **Применяется для:** Модели, сервисы, вспомогательные классы
- **Метод:** Анализ покрытия кода, тестирование всех ветвлений

### 4.3 Серый ящик (Интеграция, API)
- **Применяется для:** F-03, F-04, F-07 (проверка цепочек вызовов)
- **Метод:** Проверка API + прямой доступ к БД в тестах

## 6. Критерии входа и выхода

### 6.1 Критерии входа в тестирование
- [ ] Сборка прошла все юнит-тесты
- [ ] Smoke-тесты (критические сценарии) пройдены
- [ ] Тестовое окружение развёрнуто и доступно
- [ ] Тестовые данные подготовлены
- [ ] Тест-кейсы (не менее 80% для планируемых функций) готовы

### 6.2 Критерии выхода из тестирования
- [ ] Все запланированные тесты выполнены
- [ ] Нет открытых дефектов с приоритетом Critical и High
- [ ] Покрытие кода юнит-тестами ≥ ____% (указать целевое)
- [ ] Нагрузочное тестирование показало соответствие N-01 и N-02

Этап 4. Разработка чек-листа для Smoke-тестирования (20 минут)
Smoke-тестирование — быстрая проверка критической функциональности «на дым». Если smoke-тесты не прошли, детальное тестирование не начинается.

Задание В. Ресурсное планирование
Рассчитайте, сколько человеко-часов потребуется для тестирования релиза 1.0, если:

Ручное тестирование (T-01, T-02, T-03, T-04) — 45 тест-кейсов, каждый в среднем 5 минут

Автоматизация (регресс) — 20 автотестов, каждый 2 минуты на прогон, 30 минут на анализ

Заведение и ретест багов — 10 багов, каждый 15 минут

Формула: общее время = (ручные × время) + (автотесты × (прогон + анализ)) + (баги × время)
