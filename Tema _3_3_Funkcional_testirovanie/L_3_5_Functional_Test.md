## 1. Функциональное тестирование: основы

### 1.1 Что такое функциональное тестирование?

**Функциональное тестирование** — это проверка того, **что** система делает, без оценки того, **как** она это делает (производительность, интерфейс, безопасность — это другие типы тестирования).

> **Простое определение:** Функциональное тестирование отвечает на вопрос: «Работает ли функция так, как указано в требованиях?»

### 1.2 Что проверяем при функциональном тестировании?

| Категория | Примеры | Что проверяем |
|:---|:---|:---|
| **Точность** | Калькулятор, расчёт скидки | 2 + 2 = 4? Скидка 10% от 100 = 10? |
| **Корректность** | Форма регистрации | Нельзя зарегистрироваться с уже существующим email |
| **Полнота** | Процесс заказа | Все шаги (корзина → оплата → подтверждение) работают |
| **Обработка ошибок** | Валидация | При пустом поле «Имя» показывается сообщение |
| **Граничные случаи** | Лимиты | Максимальная сумма перевода — 1 000 000 |

### 1.3 Функциональное vs Нефункциональное тестирование

| Аспект | Функциональное | Нефункциональное |
|:---|:---|:---|
| **Что проверяем** | Что система делает | Как система это делает |
| **Пример вопроса** | Может ли пользователь войти? | Быстро ли происходит вход? |
| **Требования** | Бизнес-требования, use cases | Производительность, безопасность, удобство |
| **Кто проверяет** | QA, заказчик | QA-специалисты, DevOps |
| **Виды тестов** | Модульные, интеграционные, системные | Нагрузочные, UI/UX, безопасности |

## 2. Тестирование пользовательского интерфейса (UI-тестирование)

### 2.1 Что такое UI-тестирование?

**UI-тестирование (User Interface Testing)** — проверка того, насколько правильно и удобно пользователь взаимодействует с системой через интерфейс (кнопки, поля, формы, навигацию).

> **UI ≠ Usability.** UI-тестирование проверяет **работоспособность** элементов интерфейса (кнопка нажимается, поле заполняется), а Usability — **удобство** (понятно ли, куда нажимать).

### 2.2 Что проверяем в UI:

| Элемент | Что проверяем | Пример дефекта |
|:---|:---|:---|
| **Кнопки** | Нажимаются, ведут на правильную страницу, не активны когда нужно | Кнопка «Удалить» активна для гостя |
| **Поля ввода** | Принимают текст, валидируют, очищаются | Поле email принимает "test@" без домена |
| **Чекбоксы/радио** | Переключаются, сохраняют состояние | Выбрано два радио одновременно |
| **Выпадающие списки** | Отображают все选项, корректно выбирают | Список пуст или не скроллится |
| **Ссылки** | Ведут на правильный URL, открываются в нужном окне | Ссылка «Регистрация» ведёт на логин |
| **Сообщения об ошибках** | Появляются, понятны, исчезают при исправлении | «Ошибка 500» вместо «Неверный пароль» |
| **Навигация (меню)** | Все пункты работают, активный пункт подсвечен | Клик на «Контакты» ведёт на «О нас» |

3. Сценарии использования (Use Cases) и тестирование
3.1 Что такое сценарий использования?
Сценарий использования (use case) — это описание взаимодействия пользователя с системой для достижения конкретной цели.

Структура use case:

text
Название: _________________________
Актор: (пользователь, система, внешний сервис)
Предусловия: что должно быть правдой до начала
Основной поток (Happy path):
  1. Действие пользователя
  2. Реакция системы
  3. ...
Альтернативные потоки:
  1. ... (отклонения от основного сценария)
Постусловия: состояние системы после завершения
3.2 Пример use case: «Восстановление пароля»
markdown
**Название:** Восстановление пароля
**Актор:** Незарегистрированный пользователь (забыл пароль)
**Предусловия:** Пользователь уже зарегистрирован, но не авторизован

**Основной поток (счастливый путь):**
1. Пользователь нажимает «Забыли пароль?» на странице логина
2. Система открывает форму запроса сброса пароля (поле email)
3. Пользователь вводит email, нажимает «Отправить»
4. Система проверяет, что email существует в БД
5. Система генерирует токен, сохраняет его (срок жизни 1 час)
6. Система отправляет письмо на email со ссылкой `reset?token=...`
7. Пользователь переходит по ссылке
8. Система открывает форму нового пароля (два поля: пароль, подтверждение)
9. Пользователь вводит новый пароль, нажимает «Сбросить»
10. Система обновляет пароль в БД, удаляет токен
11. Система редиректит на страницу логина с сообщением «Пароль изменён»

**Альтернативный поток 1 (email не найден):**
4a. Система не находит email в БД
5a. Система показывает сообщение «Пользователь с таким email не найден» (но без уточнения, зачем — безопасность)
6a. Письмо не отправляется

**Альтернативный поток 2 (токен истёк):**
8a. Система проверяет токен, обнаруживает, что прошло > 1 часа
9a. Система показывает сообщение «Ссылка устарела, запросите сброс заново»
10a. Редирект на форму запроса сброса

**Постусловия:** Пароль изменён, пользователь может войти с новым паролем
3.3 Как use case превращается в тест-кейсы
Use case элемент	Тест-кейсы
Основной поток	1 тест-кейс (happy path)
Каждый альтернативный поток	1+ тест-кейсов
Предусловия	Проверка в начале теста
Постусловия	Проверка после теста
Исключения и граничные случаи	Дополнительные тест-кейсы
Из одного use case обычно получается 3-7 тест-кейсов.

3.4 Пример: тест-кейсы на основе use case «Восстановление пароля»
TC-RESET-01 (Основной поток):

Ввести существующий email → получить письмо → перейти по ссылке → ввести новый пароль → вход успешен

TC-RESET-02 (Email не найден):

Ввести несуществующий email → сообщение «Пользователь не найден» (без указания, существует ли email)

TC-RESET-03 (Повторный запрос):

Запросить сброс дважды подряд → второе письмо с новым токеном, старый токен инвалидируется

TC-RESET-04 (Истёкший токен):

Запросить сброс, подождать > 1 часа, перейти по ссылке → сообщение «Ссылка устарела»

TC-RESET-05 (Некорректный новый пароль):

Ввести пароль короче 6 символов → сообщение об ошибке, пароль не меняется

TC-RESET-06 (Несовпадение паролей):

Ввести разные значения в поля «пароль» и «подтверждение» → сообщение «Пароли не совпадают»

TC-RESET-07 (Повторное использование старого пароля):

Сбросить пароль на тот же, что был → ошибка «Новый пароль не должен совпадать со старым»

5.1 Чек-лист: что должно быть в функциональном тесте
markdown
□ Чёткое описание предусловий (что уже настроено)
□ Входные данные (точные значения)
□ Ожидаемый результат (конкретный, измеримый)
□ Постусловия (очистка данных, возврат в исходное состояние)
□ Привязка к требованию (ID требования)
□ Приоритет (Critical / High / Medium / Low)
5.2 Типичные ошибки при функциональном тестировании
Ошибка	Как избежать
Тестировать только «счастливый путь»	Всегда добавлять негативные и граничные тесты
Не проверять постусловия	Проверять БД, файлы, очередь после теста
Забывать о повторных действиях	Выполнить действие дважды (дважды нажать кнопку)
Не проверять валидацию на клиенте и сервере	Обходить клиентскую валидацию (отключить JS)
Забывать о параллельных действиях	Два пользователя одновременно
5.3 Как писать хорошие тест-кейсы для UI
Плохой тест-кейс:

Проверить, что форма логина работает

Хороший тест-кейс:

Название: Вход с валидными учетными данными
Шаги:

Открыть /login

В поле «Email» ввести user@example.com

В поле «Пароль» ввести ValidPass123

Нажать кнопку «Войти»
Ожидаемый результат: Редирект на /dashboard, в правом верхнем углу отображается имя пользователя «user@example.com»

Пример ответа к упражнению 2 (чек-лист формы создания задачи)
markdown
UI-01: Поле «Заголовок» существует, обязательно, максимальная длина 255
UI-02: Поле «Описание» существует, необязательно
UI-03: Выпадающий список «Приоритет» содержит 3 option: Низкий, Средний, Высокий
UI-04: Поле «Дедлайн» — календарь, нельзя выбрать прошедшую дату
UI-05: Кнопка «Сохранить» активна только при заполненном заголовке
UI-06: При клике на «Сохранить» без заголовка — ошибка «Заголовок обязателен»
UI-07: После успешного сохранения — редирект на список задач
UI-08: Кнопка «Отмена» ведёт на список задач без сохранения
Пример ответа к упражнению 3 (use case для добавления в корзину)
markdown
**Название:** Добавление товара в корзину
**Актор:** Пользователь (авторизованный или гость)
**Предусловия:** Товар есть в наличии, пользователь на странице товара

**Основной поток:**
1. Пользователь выбирает количество (по умолчанию 1)
2. Пользователь нажимает «В корзину»
3. Система добавляет товар в корзину (сессию или БД)
4. Система показывает уведомление «Товар добавлен», обновляет счётчик корзины

**Альтернативный поток 1 (товар закончился):**
2a. Кнопка «В корзину» неактивна или заменена на «Нет в наличии»
3a. Система не даёт добавить

**Альтернативный поток 2 (неавторизованный пользователь):**
4a. Система сохраняет товар в сессионную корзину
5a. При оформлении заказа предложит войти или зарегистрироваться

**Постусловия:** Товар в корзине, количество корзины увеличилось на 1
