## 1. Подходы к тестированию: три типа «ящиков»

Представьте, что программа — это **чёрный ящик**. Вы можете:
- **Видеть, что входит** (нажали кнопку, отправили форму).
- **Видеть, что выходит** (результат на экране, запись в БД).

Но вы **можете или не можете** видеть, что внутри (код, структура БД, логика). От этого зависит подход.

### Сравнительная таблица подходов

| Характеристика | Чёрный ящик | Белый ящик | Серый ящик |
|:---|:---|:---|:---|
| **Знание внутреннего устройства** | Нет | Да (полный доступ к коду) | Частичное (знаю API, БД, но не все детали) |
| **Кто чаще использует** | Тестировщик, заказчик, пользователь | Разработчик (юнит-тесты) | Тестировщик автотестов, интегратор |
| **Что проверяем** | Функции, требования, UI | Логику внутри методов, ветвления, циклы, граничные значения | Взаимодействие модулей, API, базы данных |
| **Пример в Laravel** | Проверить: при GET /posts выводится список статей | Проверить: метод `calculateDiscount(100)` возвращает 90 | Проверить: контроллер правильно вызывает метод репозитория и сохраняет в БД |

### Детальный разбор каждого подхода

#### 1.1 Чёрный ящик (Black Box Testing)

**Суть:** Вы — пользователь. Нажимаете кнопки, вводите данные, смотрите результат. Как устроено внутри — не важно.

**Когда применяется:**
- Функциональное тестирование
- Приёмочное тестирование (заказчик проверяет)
- Тестирование UI/UX

**Плюсы:** Не нужен доступ к коду. Может выполнять любой человек.  
**Минусы:** Не найдёт ошибки внутри логики (например, неправильный расчёт скидки, если на экране всё выглядит нормально).

**Практический пример (ваш код на Laravel):**
```php
// Требование: при POST /posts сохраняется новая статья
// Тест "чёрным ящиком" (можно сделать через браузер или Postman):

POST /posts
Content-Type: application/json

{
    "title": "Моя первая статья",
    "content": "Текст статьи..."
}

// Ожидаемый результат: HTTP 201 Created, статья появилась в списке /posts
1.2 Белый ящик (White Box Testing)
Суть: Вы видите весь код, структуру БД, маршруты. Проверяете не только результат, но и путь, которым к нему пришли.

Когда применяется:

Модульное тестирование (unit tests)

Проверка сложных алгоритмов

Анализ покрытия кода (code coverage)

Плюсы: Находит скрытые ошибки (например, неинициализированные переменные, неправильные условия).
Минусы: Требует доступа к коду и знаний программирования. Не проверяет, что на самом деле нужно пользователю.

Практический пример (юнит-тест для Laravel):

php
// Метод в контроллере или сервисе
public function calculateDiscount($total)
{
    if ($total > 1000) {
        return $total * 0.9; // 10% скидка
    } elseif ($total > 500) {
        return $total * 0.95; // 5% скидка
    }
    return $total;
}

// Тест "белым ящиком" — проверяем все ветки if
public function testDiscountCalculation()
{
    // Граничные значения
    $this->assertEquals(1000, calculateDiscount(1000)); // без скидки
    $this->assertEquals(950, calculateDiscount(1000));  // ОШИБКА? Должно быть 1000!
    $this->assertEquals(950, calculateDiscount(1000));  // Исправлено: 1000
    
    $this->assertEquals(950, calculateDiscount(1000));  // 1000 * 0.95
    $this->assertEquals(900, calculateDiscount(1000));   // 1000 * 0.9
}
1.3 Серый ящик (Grey Box Testing)
Суть: Нечто среднее. Вы знаете архитектуру, API, структуру БД, но не видите детали реализации каждого метода.

Когда применяется:

Интеграционное тестирование

Тестирование API

Тестирование баз данных

Автоматизированные тесты (когда тестировщик пишет код, но не углубляется в каждый класс)

Плюсы: Комбинирует преимущества обоих подходов. Эффективен для распределённых систем.
Минусы: Требует больше подготовки, чем чёрный ящик.

Практический пример (тест API в Laravel):

php
// Знаем: есть роут POST /api/posts, он принимает JSON, вызывает PostController@store
// Знаем: в БД есть таблица posts с полями title, content
// НЕ знаем: как именно PostService обрабатывает данные внутри

public function testStorePostViaApi()
{
    // Серый ящик: знаем API и БД, но не внутреннюю логику сервиса
    $response = $this->postJson('/api/posts', [
        'title' => 'Тестовая статья',
        'content' => 'Контент для теста'
    ]);
    
    $response->assertStatus(201);
    
    // Проверяем БД напрямую (частичное знание внутри)
    $this->assertDatabaseHas('posts', [
        'title' => 'Тестовая статья'
    ]);
}

2. Уровни тестирования (пирамида тестирования)
Теперь посмотрим, когда и что тестировать. Классическая модель — пирамида тестирования (чем ниже уровень, тем больше тестов должно быть).

text
       /\
      /  \    ← Приёмочное (мало, дорогие)
     /____\
    /      \   ← Системное (среднее количество)
   /________\
  /          \  ← Интеграционное (много)
 /____________\
/              \ ← Модульное (очень много, дешёвые)
2.1 Модульное тестирование (Unit Testing)
Что это: Проверка одного маленького компонента (метода, функции, класса) в изоляции от остальной системы.

Кто пишет: Разработчик (вы сами, когда пишете код).

Что проверяем:

Правильность расчётов

Обработку граничных значений

Поведение при null, пустых строках

Исключения (exceptions)

Пример в Laravel:

php
// Тестируем метод форматирования цены
public function testPriceFormatting()
{
    $product = new Product();
    $product->price = 1234.56;
    
    $formatted = $product->getFormattedPrice();
    
    $this->assertEquals('1 234.56 ₽', $formatted);
}
Как написать в Laravel:

bash
php artisan make:test ProductTest --unit
2.2 Интеграционное тестирование (Integration Testing)
Что это: Проверка взаимодействия между модулями. Убеждаемся, что модуль А правильно передаёт данные модулю Б.

Кто пишет: Разработчик или тестировщик-автоматизатор.

Что проверяем:

Контроллер ↔ Сервис ↔ Репозиторий

API ↔ База данных

Очереди сообщений, внешние сервисы

Пример в Laravel:

php
// Проверяем, что роут вызывает правильный метод контроллера и сохраняет в БД
public function testPostCreationFlow()
{
    // Это интеграционный тест: связка роут → контроллер → БД
    $response = $this->post('/posts', [
        'title' => 'Интеграционный тест',
        'content' => 'Проверяем всю цепочку'
    ]);
    
    $response->assertRedirect('/posts');
    $this->assertDatabaseHas('posts', ['title' => 'Интеграционный тест']);
}
2.3 Системное тестирование (System Testing)
Что это: Проверка всей системы целиком как «чёрного ящика». Все модули вместе, реальное окружение (или близкое к нему).

Кто проводит: Тестировщик (QC).

Что проверяем:

Сквозные сценарии (end-to-end)

Производительность

Нагрузка

Безопасность

Совместимость

Пример (ручной или автоматизированный):

text
Сценарий: Пользователь покупает товар

1. Зайти на сайт → Главная страница загружена
2. Найти товар "Ноутбук" → Появился в поиске
3. Добавить в корзину → Кнопка "Купить" активна
4. Оформить заказ → Форма заполнена, валидация прошла
5. Оплатить → Деньги списались, письмо пришло
6. Проверить личный кабинет → Заказ появился в истории
2.4 Приёмочное тестирование (Acceptance Testing / UAT)
Что это: Заказчик (или его представитель) проверяет, что система действительно решает его бизнес-задачи.

Кто проводит: Заказчик, бизнес-аналитик, иногда QA.

Что проверяем:

Соответствие требованиям (спецификации)

Удобство использования (можно ли реально работать)

Выполнение бизнес-целей

Пример критериев приёмки:

Требование заказчика	Как проверить	Принято / Не принято
«Пользователь может восстановить пароль за 2 минуты»	Засечь время от нажатия «Забыли пароль» до получения письма	☐
«Админ видит список заказов за сегодня»	Зайти в админку → Заказы → Фильтр «сегодня»	☐

Задание 3. Напиши тест для реального метода (10 минут)
Дано: Есть метод в сервисе (Laravel):

php
class DiscountService
{
    /**
     * Рассчитать скидку в зависимости от суммы и статуса клиента
     * @param float $total Сумма заказа
     * @param string $status Статус клиента: 'new', 'regular', 'vip'
     * @return float Итоговая сумма со скидкой
     */
    public function calculate($total, $status)
    {
        if ($status == 'vip') {
            return $total * 0.8; // 20% скидка
        } elseif ($status == 'regular') {
            if ($total > 5000) {
                return $total * 0.9; // 10% скидка
            }
            return $total * 0.95; // 5% скидка
        }
        return $total; // новым клиентам без скидки
    }
}
Задача: Напишите модульные тесты (белый ящик), чтобы покрыть все возможные ветвления. Используйте псевдокод или реальный PHPUnit синтаксис.

Подсказка: нужно проверить минимум 5 сценариев.

Пример начала:

php
public function testVipClientGets20PercentDiscount()
{
    $service = new DiscountService();
    $result = $service->calculate(1000, 'vip');
    $this->assertEquals(800, $result);
}
